# Phase 1: Baseline Validation - Context

**Gathered:** 2026-04-02
**Status:** Ready for planning

<domain>
## Phase Boundary

New main entry point (`main_swiglu_v2.py`) runs the existing fixed-tile pipeline end-to-end on BIG BOY config (256x2048x8192, tiles 16/128/32). A regression test captures ground truth outputs (CO objective, z_stop, fire counts, latency metrics) as a JSON fixture.

Additionally, a new mapping generation function is created that uses a `tile_options` format (list of candidate tile sizes per dimension), replacing the single-`tile` format. Phase 1 validates the pipeline with single-candidate tile_options matching BIG BOY defaults.

</domain>

<decisions>
## Implementation Decisions

### Main File Design
- **D-01:** New clean entry point `main_swiglu_v2.py` built from scratch using `optimize_allocation_co()` directly — no dependency on `main_swiglu_dse_single.py`
- **D-02:** CLI args with BIG BOY defaults (`seq_len=256, embedding_dim=2048, hidden_dim=8192, tiles 16/128/32`)
- **D-03:** Print key metrics to stdout AND save JSON results file to `outputs/{experiment_id}/`

### Regression Fixture
- **D-04:** JSON fixture generated by running the baseline once, then checked into the repo. Test compares a fresh run against this file.
- **D-05:** Fixture captures: CO objective value, z_stop assignments per transfer, per-transfer fire counts, total_latency, latency_per_iteration, and overlap
- **D-06:** Regression assertions use ~1e0 tolerance (a couple of cycles) on numeric values — not exact equality. This accounts for minor solver-path differences while catching real regressions.

### Test Infrastructure
- **D-07:** pytest framework with a new `tests/` directory at project root
- **D-08:** Regression test at `tests/regression/test_baseline.py`

### Mapping Source & Format
- **D-09:** New mapping generation function in a new file under `stream/inputs/aie/mapping/` (e.g., `make_swiglu_mapping_v2.py`)
- **D-10:** New function accepts a list of candidate tile sizes and produces mappings with `tile_options` format from the start (multi-candidate capable)
- **D-11:** `tile_options` format: each `intra_core_tiling` entry has `tile_options: [list]` instead of `tile: int`. Single tile = `tile_options: [value]`. No backward compatibility with old `tile` key — clean break.
- **D-12:** Phase 1 baseline calls this function with single-element tile_options matching BIG BOY defaults to validate the existing pipeline

### Output Structure
- **D-13:** Same `outputs/{experiment_id}/` pattern as existing scripts. experiment_id includes 'v2' to distinguish from old runs.

### Claude's Discretion
- JSON results file structure (key names, nesting) — as long as it captures the required metrics
- pytest configuration details (conftest.py, markers for slow tests)
- Experiment ID format for v2 runs

</decisions>

<specifics>
## Specific Ideas

- The new mapping format uses `tile_options: [list]` per dimension in `intra_core_tiling`, not `tile: int`. This is the core format change that enables the CO to select tile sizes in later phases.
- For Phase 1, the single-candidate `tile_options` must produce identical behavior to the old fixed-tile pipeline — this is the regression guarantee.
- Tolerance of ~1e0 (a couple of cycles) on numeric assertions, since even with single-candidate tile_options the solver path may differ very slightly.

</specifics>
