## Summary

Phase 1 creates a clean v2 pipeline entry point (`main_swiglu_v2.py`) and a new mapping generation function that uses a `tile_options` list format instead of a scalar `tile`. The implementation is almost entirely wiring together existing well-understood pieces: `optimize_allocation_co()`, `make_swiglu_workload()`, and a new `make_swiglu_mapping_v2()` function modelled after the existing `make_swiglu_mapping2()`.

The primary complexity is in three narrow areas: (1) understanding which fields the `MappingValidator` + `MappingFactory` chain must parse for `tile_options`, since the current schema only accepts `tile: int`; (2) computing the correct regression metrics from the `StageContext` after the CO pipeline runs, since `z_stop` variables live only inside `TransferAndTensorAllocator` during the solve and are then encoded into `scheduler.ssis` reuse levels; and (3) the pytest test infrastructure, which does not yet exist but can be set up with one conftest.py and a slow-test marker.

The BIG BOY tile configuration is confirmed: `seq_len_tile_size=16, embedding_tile_size=128, hidden_tile_size=32` with `seq_len=256, embedding_dim=2048, hidden_dim=8192`. An existing complete run for this exact config exists in `outputs/dse-20260401-tilesizes-16_128_32-whole_array_strix-swiglu_256_2048_8192-4_row_8_col/` with per-experiment subdirectories.

**Primary recommendation:** Implement the `tile_options` schema change in `MappingValidator` and `MappingFactory` first (lowest-risk, well-understood), then build `make_swiglu_mapping_v2.py`, then `main_swiglu_v2.py`, then capture the fixture, then the regression test.

### Pattern 1: optimize_allocation_co() call
**What:** Call the CO pipeline directly with hardware/workload/mapping paths and capture the returned `StageContext`.
**When to use:** All v2 entry points — this is the sole public API for CO runs.
**Example:**
```python
# Source: stream/api.py
ctx = optimize_allocation_co(
    hardware=accelerator,        # path to whole_array_strix.yaml
    workload=workload_path,      # from make_swiglu_workload()
    mapping=mapping_path,        # from make_swiglu_mapping_v2()
    experiment_id=experiment_id,
    output_path="outputs",
    skip_if_exists=False,
    enable_codegen=False,
    trace_size=1048576,
    nb_cols_to_use=8,
    npu="npu2",
)
scheduler = ctx.get("scheduler")
```

### Pattern 2: Accessing CO metrics from context
**What:** After `optimize_allocation_co()` returns, extract metrics from the `SteadyStateScheduler` stored in the context.
**When to use:** Whenever post-CO metrics are needed.
**Example:**
```python
# Source: test_co.py (existing reference)
from stream.cost_model.steady_state_scheduler import SteadyStateScheduler

scheduler: SteadyStateScheduler = ctx.get("scheduler")
latency_total        = scheduler.latency_total           # int cycles
latency_per_iteration = scheduler.latency_per_iteration  # int cycles
overlap              = scheduler.overlap_between_iterations  # int cycles
iterations           = scheduler.iterations
```

### Pattern 3: Deriving per-transfer fire counts from SSIS
**What:** After the CO solve, `scheduler.ssis` (dict keyed by `Tensor` for transfer tensors) has reuse levels baked in. Fire counts are derived as: `prod(applicable_temporal_sizes) // reuse_factor`.
**When to use:** Building the regression fixture for per-transfer fire counts.
**Example:**
```python
from math import prod
from stream.workload.node import TransferNode

fire_counts = {}
for node in scheduler.steady_state_workload.get_transfer_nodes():
    for tensor in node.outputs:
        ssis = scheduler.ssis.get(tensor, None)
        if ssis is not None:
            nb_temporal = prod(ssis.get_applicable_temporal_sizes())
            reuse = ssis.reuse_factor()
            fire_counts[node.name] = nb_temporal // reuse
```

### Pattern 4: Deriving z_stop assignments from SSIS
**What:** After the solve, `TransferAndTensorAllocator.update_tensor_steady_state_iteration_spaces()` encodes `z_stop` decisions into `scheduler.ssis` as `Reuse.REUSE` / `Reuse.NO_REUSE` flags on each `IterationVariable`. The effective stop index is the index of the last loop variable with `Reuse.REUSE`.
**When to use:** Building the regression fixture for z_stop assignments.
**Example:**
```python
from stream.workload.steady_state.iteration_space import Reuse

z_stop_assignments = {}
for node in scheduler.steady_state_workload.get_transfer_nodes():
    for tensor in node.outputs:
        ssis = scheduler.ssis.get(tensor, None)
        if ssis is not None:
            # reuse_summary() is JSON-serializable and captures the full reuse structure
            z_stop_assignments[str(tensor)] = ssis.reuse_summary()
```

### Pattern 5: CO objective value
**What:** The Gurobi model objective value is accessible as `model.ObjVal` on the allocator, but the allocator (`tta`) is a local variable inside `SteadyStateScheduler.run()` and is NOT stored as an attribute. The best available proxy is `scheduler.latency_total` (the `total_latency` component of the objective). The full objective = `total_latency + max_dma_in + max_dma_out`, but `max_dma_in/out` are not individually surfaced post-solve.

**Implication for fixture:** Use `scheduler.latency_total` as the primary "CO objective value" metric for the regression fixture. This is the dominant term and what the user cares about for regression.

**Alternative (HIGH confidence):** Extend `SteadyStateScheduler` to store the final `model.ObjVal` from `tta.model.ObjVal` before `tta.solve()` returns, and expose it as `scheduler.co_objective`. This requires a 3-line change to `steady_state_scheduler.py`. Recommended — makes the intent explicit.

### Pattern 6: tile_options format in mapping YAML
**What:** The new format replaces `tile: int` with `tile_options: [int]` in each `intra_core_tiling` entry.
**When to use:** All v2 mapping generation functions.
**Example (YAML output):**
```yaml
fused_groups:
- name: Fused_Group_1
  layers: [Gemm_Left, Gemm_Right, Silu, Elt_Mul]
  intra_core_tiling:
  - dim: Gemm_Left.D1
    tile_options: [128]     # was: tile: 128
  - dim: Gemm_Left.D2
    tile_options: [32]      # was: tile: 32
  - dim: Gemm_Left.D0
    tile_options: [16]      # was: tile: 16
```

### Pattern 7: MappingValidator schema for tile_options
**What:** The current `SCHEMA_FUSED_GROUP` in `MappingValidator` only accepts `tile: int`. It must be extended to accept `tile_options: list[int]` as an alternative.
**Key constraint:** `MappingFactory._convert_intra_core_tiling_entry()` currently reads `int(entry["tile"])`. For Phase 1 (single-element lists), the factory should extract the single value from `tile_options`.

**Required changes (two files):**
1. `stream/parser/mapping_validator.py` — add `tile_options` as an alternative key in `SCHEMA_FUSED_GROUP`
2. `stream/parser/mapping_factory.py` — update `_convert_intra_core_tiling_entry()` to handle either `tile` (old) or `tile_options` (new, extract first element for Phase 1)

### Anti-Patterns to Avoid
- **Reading tile_options index beyond 0 in Phase 1:** For Phase 1, `tile_options` is always a single-element list. Extracting `entry["tile_options"][0]` is correct. Do NOT try to handle multi-element lists in this phase.
- **Storing tta outside scheduler:** The `TransferAndTensorAllocator` is large and complex. Do not store it as a persistent attribute unless needed — it holds the Gurobi model with binary variables.
- **Depending on main_swiglu_dse_single.py:** Decision D-01 prohibits this. `main_swiglu_v2.py` is built from scratch.
- **Using model.ObjVal after the fact:** The Gurobi model is only valid during `tta.solve()`. The CO objective value must be captured inside that call or via the surface attributes on scheduler.

## Common Pitfalls

### Pitfall 1: MappingValidator rejects tile_options key
**What goes wrong:** `MappingValidator._validate_positive_tiling_values()` iterates fused_group entries and reads `entry.get("tile", 0)` — if `tile` key is absent (because only `tile_options` is present), it gets `0` and fails the positive-int check.
**Why it happens:** Current schema strictly requires `tile: int`; there is no fallback.
**How to avoid:** Update both the Cerberus schema (allow `tile_options` list) AND the `_validate_positive_tiling_values()` method to check `tile_options` when `tile` is absent.
**Warning signs:** `ValueError: Failed to validate user provided mapping` during `MappingParserStage`.

### Pitfall 2: MappingFactory reads tile key directly
**What goes wrong:** `MappingFactory._convert_intra_core_tiling_entry()` calls `int(entry["tile"])` — this raises `KeyError` when only `tile_options` is present.
**Why it happens:** Factory mirrors the old schema exactly.
**How to avoid:** Change factory to: `tile = entry.get("tile") or entry["tile_options"][0]` (read old key first for backward compatibility, fall back to first element of tile_options).
**Warning signs:** `KeyError: 'tile'` during mapping parsing.

### Pitfall 3: CO objective value inaccessible post-solve
**What goes wrong:** Attempting to access `tta.model.ObjVal` outside of `SteadyStateScheduler.run()` fails because `tta` is a local variable.
**Why it happens:** The scheduler does not save the allocator as an instance attribute.
**How to avoid:** Either (a) add `self.co_objective = tta.model.ObjVal` to `SteadyStateScheduler.run()` right after `tta.solve()`, or (b) use `scheduler.latency_total` as the proxy metric. Option (a) is recommended.
**Warning signs:** `AttributeError: 'SteadyStateScheduler' has no attribute 'co_objective'`.

### Pitfall 4: z_stop data is only in tensor-keyed SSIS, not transfer-node SSIS
**What goes wrong:** `scheduler.ssis` is keyed by both `TransferNode` objects AND `Tensor` objects. The z_stop reuse level is embedded in the **tensor-keyed** entries (set by `update_tensor_steady_state_iteration_spaces()`), not the transfer-node-keyed entries.
**Why it happens:** `TransferAndTensorAllocator` operates on tensors as decision-variable keys; the per-tensor SSIS carries the reuse decision.
**How to avoid:** Iterate over transfer nodes, then iterate their `outputs` (tensors), then access `scheduler.ssis[tensor]`.
**Warning signs:** Missing z_stop data if iterating `scheduler.ssis` by TransferNode key only.

### Pitfall 5: Regression test is slow (CO solver runs full Gurobi)
**What goes wrong:** Running the regression test in CI takes several minutes because it runs the full CO pipeline including Gurobi.
**Why it happens:** The BIG BOY config is a large problem.
**How to avoid:** Mark the test with `@pytest.mark.slow` and configure pytest to skip slow tests by default (`pytest -m "not slow"` for fast runs; `pytest -m slow` for full regression). Document this in conftest.py.
**Warning signs:** CI timeout if slow marker is not applied.

### Pitfall 6: Fixture reuse_summary() output depends on internal object repr
**What goes wrong:** If `str(tensor)` or `str(iv.dimension)` uses Python object ids or memory addresses, the fixture is not stable across runs.
**Why it happens:** Custom `__repr__` or fallback `object.__repr__`.
**How to avoid:** Use `node.name` + explicit tensor index as the fixture key, not `str(tensor)`.
**Warning signs:** Fixture keys look like `<stream.workload.node.Tensor object at 0x...>`.

### Accessing all CO pipeline metrics from context
```python
# Source: test_co.py + stream/cost_model/steady_state_scheduler.py
from stream.cost_model.steady_state_scheduler import SteadyStateScheduler
from stream.workload.node import TransferNode
from math import prod

ctx = optimize_allocation_co(
    hardware=accelerator,
    workload=workload_path,
    mapping=mapping_path,
    experiment_id=experiment_id,
    output_path="outputs",
    skip_if_exists=False,
    enable_codegen=False,
    trace_size=1048576,
    nb_cols_to_use=8,
    npu="npu2",
)

scheduler: SteadyStateScheduler = ctx.get("scheduler")

# Latency metrics (directly available)
total_latency = scheduler.latency_total              # int cycles
latency_per_iter = scheduler.latency_per_iteration   # int cycles
overlap = scheduler.overlap_between_iterations       # int cycles

# Per-transfer fire counts (derived from SSIS after solve)
fire_counts = {}
for node in scheduler.steady_state_workload.get_transfer_nodes():
    ssis_for_node = scheduler.ssis.get(node, None)
    if ssis_for_node is not None:
        nb = prod(ssis_for_node.get_applicable_temporal_sizes())
        reuse = ssis_for_node.reuse_factor()
        fire_counts[node.name] = nb // reuse if reuse > 0 else nb

# z_stop (reuse) assignments - JSON-serializable via reuse_summary()
z_stop = {}
for node in scheduler.steady_state_workload.get_transfer_nodes():
    for i, tensor in enumerate(node.outputs):
        ssis_for_tensor = scheduler.ssis.get(tensor, None)
        if ssis_for_tensor is not None:
            z_stop[f"{node.name}__out{i}"] = ssis_for_tensor.reuse_summary()
```

## Open Questions

1. **How to surface CO model.ObjVal from scheduler**
   - What we know: `tta.model.ObjVal` is the full Gurobi objective value. `scheduler.latency_total` = the `total_latency` variable component.
   - What's unclear: Whether the fixture should capture the full objective (total_lat + dma) or just latency_total. `max_core_dma_in` and `max_core_dma_out` are also vars in the model but not exposed post-solve.
   - Recommendation: Add `self.co_objective = tta.model.ObjVal` to `SteadyStateScheduler.run()` after `tta.solve()`. This is a 1-line addition and makes the fixture unambiguous.

2. **Fire count source: TransferNode SSIS vs. Tensor SSIS**
   - What we know: `scheduler.ssis` has entries for both TransferNodes and Tensors. Reuse levels are set on tensor-keyed entries.
   - What's unclear: Whether TransferNode-keyed entries have accurate `applicable_temporal_sizes` for fire count computation, or whether only tensor-keyed entries should be used.
   - Recommendation: Use tensor-keyed entries (as `MemoryAccessesEstimationStage` does) for accuracy. Cross-check with `memory_accesses_estimation.py:83` pattern: `prod(ssis.get_applicable_temporal_sizes()) // ssis.reuse_factor()`.
